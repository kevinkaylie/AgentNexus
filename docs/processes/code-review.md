# 代码评审流程

## 概述

本文档定义 AgentNexus 项目的代码评审流程。所有功能代码在合并前必须经过代码评审，评审通过后开发 Agent 按意见修复 Bug 并补充测试用例。

## 流程图

```
开发完成 → 提交评审 → 评审 Agent 审查 → 反馈记录
                                            │
                          ┌─────────────────┴──────────────────┐
                          ▼ 有阻塞性问题                        ▼ 无阻塞性问题
                   开发 Agent 修复                            批准合并
                          │
                          ▼
                   修复确认（评审 Agent）
                          │
                          ▼
                        批准合并
```

## 触发条件

以下情况**必须**走代码评审：

| 情况 | 说明 |
|------|------|
| 新功能实现完成 | wip.md 中对应条目状态变为"待评审" |
| 安全相关变更 | 涉及密码学、私钥、Token、签名验证的任何改动 |
| 接口变更 | 新增或修改 HTTP 端点、修改请求/响应格式 |
| 重构 | 影响多个模块的结构性改动（如 ADR-009 DID 注册表重构） |

以下情况**可跳过**代码评审（走快速路径）：

- 文档修改（`.md` 文件）
- 测试用例补充（不涉及功能代码）
- 配置文件调整（不影响运行逻辑）

## 五阶段流程

### 阶段 1：提交评审

**执行者：** 开发 Agent

1. 将 wip.md 中对应条目状态更新为"待评审"
2. 通知评审 Agent，说明：
   - 变更范围（涉及哪些文件）
   - 关联 ADR 编号
   - 已通过的测试情况

### 阶段 2：评审 Agent 审查

**执行者：** 评审 Agent

按以下检查清单逐项审查：

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | ADR 合规 | 代码实现是否与关联的已采纳 ADR 一致 |
| 2 | 安全性 | 私钥不出 Daemon、签名验证正确、Token 鉴权完整、无明文敏感信息 |
| 3 | 正确性 | 逻辑 Bug、边界条件、异常处理是否完整 |
| 4 | 测试覆盖 | 新功能是否有对应测试用例，关键路径是否覆盖 |
| 5 | 接口一致性 | HTTP 端点格式是否与 `docs/api-reference.md` 一致 |
| 6 | 代码规范 | 遵循 `CLAUDE.md` 和 `agent_net/CLAUDE.md` 中的编码规范 |

### 阶段 3：反馈记录

**执行者：** 评审 Agent

评审意见分三类：

| 类型 | 说明 | 开发 Agent 是否必须处理 |
|------|------|----------------------|
| 阻塞性（Bug） | 功能错误、安全漏洞、运行时崩溃 | **必须修复后才能合并** |
| 建议性 | 代码质量、可读性、潜在风险 | 建议处理，可在后续迭代修复 |
| 信息性 | 备注、说明，不需要改动 | 无需处理 |

评审意见写入对应的代码评审记录（见下方格式）。

### 阶段 4：开发 Agent 修复

**执行者：** 开发 Agent

1. 按评审意见修复所有**阻塞性 Bug**
2. 为每个 Bug 补充或修改对应的**测试用例**，确保修复有测试覆盖
3. 本地运行全量测试，确认通过
4. 将修复内容和新增测试用例反馈给评审 Agent

**修复要求：**
- 每个阻塞性 Bug 必须有对应测试用例（防止回归）
- 建议性问题如果本次一并修复，需在反馈中注明

### 阶段 5：修复确认与批准

**执行者：** 评审 Agent

1. 确认所有阻塞性问题已修复
2. 确认新增测试用例覆盖了修复点
3. 批准合并，开发 Agent 将 wip.md 条目状态更新为"✅ 完成"

## 代码评审记录格式

评审完成后，在 `docs/devlog.md` 中追加记录：

```markdown
## 代码评审 — {功能名称} ({日期})

**关联 ADR：** ADR-XXX
**评审者：** 评审 Agent
**结果：** 批准 / 条件批准

### 阻塞性问题

| # | 文件 | 问题描述 | 修复状态 |
|---|------|---------|---------|
| 1 | path/to/file.py | 问题描述 | ✅ 已修复 |

### 建议性问题

| # | 文件 | 问题描述 | 处理状态 |
|---|------|---------|---------|
| 1 | path/to/file.py | 问题描述 | 本次修复 / 后续迭代 |

### 新增测试用例

| 测试文件 | 测试场景 | 覆盖的 Bug |
|---------|---------|-----------|
| tests/test_xxx.py | 场景描述 | Bug #1 |
```

## 与设计评审流程的衔接

```
ADR 批准（设计评审流程）
        │
        ▼
  设计答疑（ADR-009 答疑记录）
        │
        ▼
  开发实现（开发 Agent）
        │
        ▼
  代码评审（本流程）
        │
        ▼
  测试验证（测试 Agent）
        │
        ▼
  合并 + CHANGELOG 更新
```

## 紧急修复快速路径

安全漏洞或线上 Bug 修复可走快速路径：

1. 开发 Agent 修复并补充测试
2. 评审 Agent 仅审查安全性和正确性（跳过完整检查清单）
3. 合并后在 `docs/devlog.md` 补充评审记录
