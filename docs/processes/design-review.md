# 设计评审流程

## 概述

本文档定义 AgentNexus 项目的设计评审流程。所有 ADR 和重大设计变更必须经过此流程审查后才能标记为"已采纳"。

## 六阶段流程

```
提交评审请求 → 评审审查 → 反馈记录 → 修改确认 → 批准/驳回
                                                    │
                                                    ▼ 批准后
                                             设计答疑（新增）
                                                    │
                                                    ▼ 答疑完成
                                             交付开发实现
```

### 阶段 1：提交评审请求

**执行者：** 设计 Agent

- 将 ADR 草稿或设计文档提交给评审 Agent
- 提交内容必须包含：背景说明、决策内容、影响分析、替代方案对比
- ADR 状态标记为"提议"

### 阶段 2：评审 Agent 审查

**执行者：** 评审 Agent

按以下检查清单逐项审查：

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | 架构一致性 | 设计是否与 `docs/architecture.md` 中的现有架构一致 |
| 2 | ADR 遵循 | 是否与 `docs/adr/` 中已采纳的决策冲突 |
| 3 | 安全性考量 | 私钥隔离、签名验证、访问控制、加密通信是否符合规范 |
| 4 | 测试覆盖 | 新功能是否有对应的测试用例要求 |
| 5 | 跨团队接口影响 | 是否影响 Giskard/OATR/QNTM WG 等合作方的接口契约 |
| 6 | 文档同步 | `docs/wip.md` 中"待同步文档"列是否已标明受影响的文档，影响范围章节列出的文档是否已更新 |

### 阶段 3：反馈记录

**执行者：** 评审 Agent

- 评审意见记录在对应的 ADR 或设计文档中
- 反馈分类：阻塞性问题（必须修改）、建议性问题（可选修改）、信息性备注

### 阶段 4：修改确认

**执行者：** 设计 Agent → 评审 Agent

- 设计 Agent 根据反馈修改文档
- 评审 Agent 确认阻塞性问题已解决

### 阶段 5：批准或驳回

**执行者：** 评审 Agent

- **批准**：ADR 状态更新为"已采纳"，进入阶段 6
- **驳回**：返回阶段 1，附驳回理由

### 阶段 6：设计答疑

**执行者：** 开发 Agent → 设计 Agent

**触发条件：** ADR 被批准后、开发实现开始前（或实现过程中遇到设计歧义时）。

**流程：**

1. 开发 Agent 阅读已批准的 ADR 和设计文档
2. 针对实现层面的疑问，向设计 Agent 提交答疑请求
3. 设计 Agent 回复澄清，必要时补充设计细节
4. 答疑内容记录在对应 ADR 底部的"答疑记录"章节（格式见下方）
5. 如果答疑过程中发现设计需要修改，设计 Agent 更新 ADR 并注明修改原因

**答疑范围（允许）：**
- 实现细节澄清（"content 加密前需要 json.dumps 吗？"）
- 边界条件确认（"轮询间隔退避到 30s 后，Daemon 恢复时怎么检测？"）
- 性能可行性讨论（"批量注册 1020 条，Redis pipeline 还是逐条？"）
- 与现有代码的兼容性确认（"现有 storage.py 的 messages 表需要 migration 吗？"）

**答疑范围（不允许）：**
- 推翻已批准的架构决策（需要走新 ADR 流程）
- 绕过评审直接修改设计（设计变更必须经评审确认）

**答疑记录格式：**

在 ADR 底部追加：

```markdown
## 答疑记录

| 日期 | 提问者 | 问题 | 回复 | 是否触发设计变更 |
|------|--------|------|------|----------------|
| YYYY-MM-DD | 开发 Agent | 问题描述 | 设计 Agent 回复 | 否 / 是（已更新 §X） |
```

**退出条件：** 开发 Agent 确认无更多疑问，或所有阻塞性疑问已澄清。答疑阶段不设硬性时间限制，但建议在 ADR 批准后 1-2 个工作周期内完成。

---

## 紧急变更快速路径

当变更紧急且影响范围有限时：

1. 跳过完整评审流程
2. 评审 Agent 在 PR 中直接审查
3. 变更合并后，设计 Agent 事后补充 ADR
4. 评审 Agent 在下一个评审周期中追溯确认

**适用条件：** 仅限 bug 修复、安全补丁或不影响架构的小范围变更。新增模块、接口变更、密码学变更不适用快速路径。

## 评审结果记录

评审完成后，在对应 ADR 底部追加评审记录：

```markdown
## 评审记录

| 日期 | 评审者 | 结果 | 备注 |
|------|--------|------|------|
| YYYY-MM-DD | 评审 Agent | 批准/驳回 | 简要说明 |
```
